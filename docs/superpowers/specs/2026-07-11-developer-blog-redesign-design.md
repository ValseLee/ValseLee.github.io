# 개발자 블로그 리디자인 설계

## 목표

기존 정적 개발자 블로그를 `ph.demiladehq.com`의 편집형 시각 문법으로 재구성한다. 초대형 제목, 번호가 붙은 섹션, 촘촘한 그리드, 흑백 타이포그래피를 차용하되 콘텐츠와 구조는 Celan의 개발자 블로그에 맞춘다.

배포 결과는 계속 GitHub Pages에서 동작하는 정적 사이트여야 한다. 글과 프로필 콘텐츠를 편집하는 기능은 기존 아티클 대시보드처럼 `127.0.0.1`에서만 실행하며 배포 사이트에 Admin 경로를 만들지 않는다.

## 공개 사이트 구조

홈은 다음 순서의 긴 편집형 페이지로 구성한다.

1. Hero: 이름 또는 사이트명, 개발자 역할, 블로그 설명을 초대형 제목과 짧은 소개문으로 표시한다.
2. About: 소개와 작업 관점 또는 원칙을 표시한다.
3. Expertise: 전문 분야를 그룹과 항목으로 나눠 표시한다.
4. Articles: 모든 게시글을 카드형 그리드로 표시한다. 이미지 대신 제목, 날짜, 카테고리, 태그와 본문 첫 10문장을 제공한다.
5. Experience: 경력과 활동을 기간, 조직, 역할, 설명으로 표시한다.
6. Contact: 이메일과 소셜 링크, 저작권 문구를 표시한다.

기존 `/archive`, `/categories`, `/translations`, `/graph`, `/posts/[slug]`, `/translations/[slug]`, `/about` 라우트와 연결 기능은 유지한다. 공통 헤더와 각 페이지의 타이포그래피, 선, 간격, 색상을 홈과 같은 시각 언어로 통일한다.

## 시각 방향

- 밝은 종이색 배경과 검은 전경색을 기본으로 한다.
- 제목은 화면 너비를 적극적으로 사용하는 굵은 산세리프체로 구성한다.
- 데스크톱에서는 12열 기반의 촘촘한 배치를, 작은 화면에서는 한 열의 읽기 쉬운 흐름을 사용한다.
- 섹션 번호, 대문자 라벨, 얇은 구분선으로 정보 계층을 만든다.
- 이미지, 장식용 SVG, 새로운 아이콘 패키지는 추가하지 않는다.
- 링크와 카드에는 색상보다 밑줄, 반전, 이동 같은 단순한 상태 변화를 사용한다.
- `prefers-reduced-motion`을 존중하고 키보드 포커스를 명확히 표시한다.

## 콘텐츠 모델

사이트에서 편집할 프로필 콘텐츠는 `content/site.json` 한 파일에 저장한다.

- `identity`: 이름, 역할, 사이트 제목, 소개문
- `about`: 업데이트 표기, 소개 본문, 작업 관점, 원칙
- `expertise`: 그룹명과 항목 배열
- `experience`: 기간, 조직, 역할, 설명 배열
- `contact`: 이메일, 소셜 링크, 저작권 문구

공개 페이지는 이 파일을 서버 컴포넌트에서 직접 읽는다. 별도 데이터베이스, API, 런타임 상태는 만들지 않는다.

게시글의 10문장 미리보기는 MDX 본문에서 코드 블록과 Markdown 표식을 제거한 뒤 문장 종결 부호를 기준으로 최대 10문장을 선택한다. 문장 종결 부호가 부족한 글은 남은 본문을 자연스럽게 잘라 표시한다. 이 로직은 `lib/posts.ts`의 게시글 읽기 흐름에 한 번만 둔다.

## 로컬 Admin

기존 `npm run write` 대시보드를 유지하고 같은 로컬 서버에 사이트 콘텐츠 편집 화면을 추가한다.

- 게시글 편집 기능과 사이트 콘텐츠 편집 기능을 명확한 탭 또는 섹션으로 구분한다.
- 사이트 편집 화면에서 Hero, About, Expertise, Experience, Contact의 모든 값을 수정할 수 있다.
- 반복 항목은 새 UI 프레임워크 없이 브라우저 기본 입력 요소와 추가·삭제 버튼으로 편집한다.
- 저장 요청은 로컬 서버가 입력을 검증한 뒤 `content/site.json`을 원자적으로 교체한다.
- 서버는 계속 `127.0.0.1`에만 바인딩한다.
- 공개 Next.js 앱에는 `/admin` 라우트나 쓰기 API를 추가하지 않는다.

## 데이터 흐름과 오류 처리

공개 사이트는 빌드 시 `content/site.json`과 MDX 파일을 읽어 정적 HTML을 생성한다. 콘텐츠 파일이 없거나 구조가 잘못되면 조용히 빈 페이지를 만들지 않고 빌드가 이해 가능한 오류와 함께 실패하도록 한다.

Admin은 저장 전에 필수 문자열, 배열 구조, URL 형식을 검증한다. 검증 실패 시 파일을 변경하지 않고 화면에 오류를 표시한다. 저장은 임시 파일을 작성한 뒤 rename하는 방식으로 수행해 중간 실패 시 기존 파일을 보존한다.

## 검증

- 사이트 콘텐츠 파싱과 저장을 검증하는 작은 Node 테스트를 추가한다.
- 게시글 미리보기가 최대 10문장을 반환하는지 검증한다.
- 기존 아티클 대시보드 테스트를 유지한다.
- `npm run test:article-dashboard`, 콘텐츠 검증, 린트, 프로덕션 빌드를 실행한다.
- 브라우저에서 데스크톱과 모바일 홈, 게시글 상세, 로컬 Admin의 읽기·편집·저장 흐름을 확인한다.

## 범위에서 제외

- 배포된 Admin, 사용자 인증, 데이터베이스, 외부 CMS
- 이미지 기반 작품 카드와 이미지 업로드
- 레퍼런스 사이트의 다국어 복제
- 분석 도구, 검색, 댓글, 뉴스레터
- 레퍼런스 사이트의 코드나 고유 자산 복사
